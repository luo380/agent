第 1 周：FAISS 向量索引集成学习规划

这份规划只解决一个问题：
把你当前项目里的“暴力向量检索”升级成“基于 FAISS 的索引召回”。

这一周先不做 Agent，不做 LangGraph，不做复杂工作流。
目标很聚焦：
- 理解为什么你现在的检索会越来越慢
- 学会 FAISS / HNSW 的基本原理
- 在你自己的项目里完成第一版索引化召回
- 保留你现有的 rerank 逻辑，不一次性重构过头


一、本周为什么该学这个

结合你现在的项目实现，你的检索链路大致是这样的：

1. 文档上传后，把 chunk embedding 存到数据库
2. 用户提问时，先把问题做 embedding
3. 再把当前用户的所有 chunks 全部查出来
4. 把每条 chunk 的 embedding_json 反序列化
5. 一条条做 cosine similarity
6. 取 top_k 候选
7. 再做 rerank

这个方案在学习阶段是很好的，因为你能真正理解向量检索的底层。
但它有 3 个明显问题：

1. 数据一多就慢
因为你每次都要把所有 chunk 拉出来，再在 Python 里逐条计算相似度。

2. CPU 和内存浪费
embedding 存成 JSON 字符串后，每次检索都要重复 parse。

3. 后面不好扩展
当知识库从几百条 chunk 变成几万条 chunk 时，暴力搜索会成为明显瓶颈。

所以这一周学 FAISS，不是“为了炫技”，而是为了让你的 RAG 从“能跑”走向“更像真实系统”。


二、本周结束时你应该达到的结果

学完这一周，你应该能做到：

1. 说清楚什么是暴力搜索
2. 说清楚什么是向量索引
3. 说清楚 FAISS 是干什么的
4. 说清楚 HNSW 和 Flat 的区别
5. 在本地用 FAISS 构建一份向量索引
6. 在项目里把“粗召回”替换成 FAISS 检索
7. 保留现有 rerank，不影响回答质量链路
8. 能对比替换前后的耗时和效果

一句话说，就是：
从“手写暴力检索 RAG”升级成“带近似向量索引的 RAG”。


三、本周你要掌握的核心概念 

1. 暴力搜索
定义：
把所有向量都拿出来，逐个计算相似度，再排序。

优点：
- 最容易理解
- 结果精确
- 适合小数据集和教学

缺点：
- 数据一大就慢
- 每次都要全量扫描


2. 向量索引
定义：
提前把向量组织成适合搜索的数据结构，查询时不用再扫全量。

你要理解的重点：
- 向量索引不是“更准确”
- 向量索引主要解决“更快”
- 有些索引是精确检索
- 有些索引是近似检索


3. FAISS
定义：
Facebook AI Research 开源的向量相似度搜索库。

这一周你要掌握的不是全部 FAISS，而是最关键的两类：
- IndexFlatIP / IndexFlatL2
- IndexHNSWFlat


4. Flat Index
含义：
虽然名字叫索引，但本质仍然接近精确搜索，只是把向量数据结构化存起来，由 FAISS 负责高效计算。

你要知道：
- Flat 是最适合入门的
- 它帮你先学会“怎么接 FAISS”
- 它不一定解决所有性能问题


5. HNSW
全称：
Hierarchical Navigable Small World

你要掌握的不是复杂数学，而是这几个点：
- 它是一种 ANN 近似最近邻方法
- 它比全量扫描快很多
- 它牺牲一部分精确性，换更高速度
- 常见参数包括图连接度和搜索深度


6. ANN
全称：
Approximate Nearest Neighbor

重点理解：
- 不一定返回“绝对最相近”的结果
- 但通常返回“足够相近”的结果
- 对 RAG 的粗召回非常适合


7. Inner Product / Cosine Similarity / L2
你这一周一定要搞清楚：
- 余弦相似度常见于 embedding 检索
- FAISS 常用 IP 或 L2
- 如果要用 IP 模拟 cosine similarity，通常需要先做向量归一化

这块非常关键。
如果这里没理解清楚，后面你会出现：
- 为什么我接了 FAISS，结果排序变了
- 为什么相关 chunk 变少了
- 为什么看起来命中率下降了


四、结合你项目，这一周的正确改造边界

这一周只改“粗召回层”，不要一次动整条链。

建议保留：
- 现有 embedding 生成逻辑
- 现有 KnowledgeChunks 数据表
- 现有 rerank_chunks()
- 现有 RAG prompt
- 现有 citations 逻辑

优先替换：
- search_similar_chunks_by_embedding() 内部的暴力扫描部分

为什么这样最合适：
- 改动小
- 风险低
- 好验证
- 你能清楚看出“索引替换”的价值


五、本周学习顺序

我建议你按 7 天来学。
如果你是兼职节奏，也可以把 7 天拉成 10 到 14 天。


第 1 天：把当前检索链路完全看懂

目标：
先彻底搞懂你现在的暴力检索，不要急着装库。

今天要做的事：

1. 重新读一遍当前检索代码
重点看：
- core/service/retrieval.py
- core/service/rag.py
- api/routes/knowledge.py

2. 你要能画出这条链路：
chunk 入库
-> embedding_json 存库
-> query embedding
-> load_user_chunks()
-> parse_embedding()
-> cosine_similarity()
-> search_similar_chunks_by_embedding()
-> rerank_chunks()

3. 记录现状问题
至少写出这几个点：
- 检索是不是每次都全量遍历
- embedding 是否每次都重新反序列化
- 当前 top_k 的性能瓶颈在哪
- rerank 前后的边界在哪

今天的产出：
- 一张你自己的“当前检索链路图”
- 一份“为什么现在需要索引”的简短说明


第 2 天：学习 FAISS 基础概念

目标：
知道 FAISS 的基本使用方式和常见索引类型。

今天要学的内容：

1. FAISS 是什么
2. IndexFlatL2 是什么
3. IndexFlatIP 是什么
4. cosine similarity 和 inner product 的关系
5. 什么叫向量归一化

你今天必须吃透这句话：
如果 embedding 检索要近似 cosine similarity，那么通常先把向量做 L2 normalize，再用 inner product 检索。

今天的产出：
- 你能口头解释 Flat 和 brute force 的关系
- 你知道为什么“接 FAISS 后分数可能变”


第 3 天：本地写一个最小 FAISS Demo

目标：
脱离项目，先跑最小实验。

建议做法：

1. 准备几条简单文本
2. 生成几组假的向量，或者用真实 embedding
3. 用 numpy 组织成二维数组
4. 构建一个 IndexFlatIP
5. add 向量
6. search top_k
7. 验证返回的 id 和分数

今天不要急着接数据库。
先把这几个概念跑顺：
- 维度一致
- dtype 要对
- add 和 search 的入参格式
- index 返回的是距离和 id

今天的产出：
- 一个独立可跑的 FAISS 最小 demo
- 你知道 search 返回结果如何映射回原始 chunk


第 4 天：学习 HNSW 原理和使用场景

目标：
理解为什么不只是 Flat，还要学 HNSW。

今天要学的点：

1. Flat 和 HNSW 的区别
2. 精确搜索和近似搜索的区别
3. HNSW 为什么快
4. HNSW 在什么场景适合 RAG 粗召回
5. 常见参数大致影响什么

你不需要深究公式，
但必须能说清楚：
- Flat 更适合入门和对照实验
- HNSW 更适合真实检索加速

今天的产出：
- 一份你自己的对比表

建议表格至少包含：
- 检索方式
- 是否近似
- 速度
- 精度
- 集成难度
- 适合当前项目的阶段


第 5 天：设计你项目里的第一版接入方案

目标：
不是直接开改，而是先设计。

今天要想清楚 5 个问题：

1. 索引建在哪里
可选思路：
- 进程启动时扫描数据库重建
- 每次上传文档后增量重建
- 先做内存级索引，后面再持久化

2. chunk 元数据怎么回查
因为 FAISS 只存向量和内部 id，
你还需要一层映射：
- faiss row id -> chunk_id / document_id / metadata

3. 多用户怎么隔离
你现在是多用户知识库，
要考虑：
- 每个用户一个索引
- 或一个总索引加 user_id filter

对你现在阶段，更推荐：
每个用户一个索引，逻辑更直观。

4. 索引什么时候更新
至少要考虑：
- 新文档上传完成后
- 文档删除后
- 文档重新切块后

5. 出错时如何降级
如果索引不可用，是否回退到暴力搜索？

今天的产出：
- 一份你的第一版接入设计说明


第 6 天：开始项目级改造

目标：
把项目里的粗召回改成“优先 FAISS，必要时降级暴力搜索”。

建议改造顺序：

1. 先抽出索引层服务
建议新增一个独立服务文件，例如：
- core/service/vector_index.py

建议职责：
- 构建索引
- 向量归一化
- add 向量
- search top_k
- 维护 row id 和 chunk 的映射

2. 不要一上来改数据库表
你这周重点是先跑通检索替换。
数据库结构可以暂时不变。

3. 在 retrieval.py 里保留旧逻辑
建议改成：
- 先尝试走 FAISS
- 如果失败，再 fallback 到现有 brute force

4. 保留 rerank
也就是：
FAISS 负责召回候选
rerank_chunks() 负责精排

今天的产出：
- 项目里第一版 FAISS 检索链路能跑通


第 7 天：验证效果，写总结

目标：
不是“感觉能跑了”，而是验证有价值。

今天要做的验证：

1. 性能对比
至少比较：
- 10 条 chunk
- 100 条 chunk
- 1000 条 chunk

即使暂时没有那么多真实数据，也可以先构造实验数据。

2. 结果对比
比较：
- 暴力检索 top_k
- FAISS Flat top_k
- FAISS HNSW top_k

看：
- 命中是否接近
- 召回是否明显偏移

3. 系统稳定性
验证：
- 某个用户没有 chunk 时能否安全返回
- 向量维度不一致时是否报错清晰
- 索引失效时是否能降级

4. 写学习总结
至少写出这几条：
- 我原来的暴力检索瓶颈是什么
- FAISS 解决了什么
- Flat 和 HNSW 有什么区别
- 为什么现在我只替换粗召回，不动 rerank

今天的产出：
- 一份本周总结
- 一份性能和效果对比记录


六、本周最推荐的技术落地路径

顺序建议如下：

第 1 步：
先接 FAISS Flat

原因：
- 概念最清晰
- 好对照
- 出问题最好排查

第 2 步：
验证和暴力搜索结果是否大体一致

第 3 步：
再尝试 HNSW

第 4 步：
比较 Flat 和 HNSW 的速度与命中差异

这个顺序比“直接上 HNSW”稳很多。


七、本周不要做什么

为了避免你学偏，这周不要顺手做这些：

1. 不要同时改 rerank
不然你到最后分不清是召回变了，还是精排变了。

2. 不要同时改 prompt
不然最终回答变化会掩盖检索变化。

3. 不要一上来接 pgvector / Milvus
先把 FAISS 和索引思路搞明白。

4. 不要一上来就做分布式、多机、复杂持久化
你当前项目阶段还不需要。

5. 不要急着把数据库 schema 全改掉
这一周核心是“学习并跑通索引替换”。
